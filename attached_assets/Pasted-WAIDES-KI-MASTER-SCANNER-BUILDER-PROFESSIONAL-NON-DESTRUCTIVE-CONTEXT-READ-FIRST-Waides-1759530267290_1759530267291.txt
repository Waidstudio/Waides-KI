BUSINESS RULES & BEHAVIOR (IMPLEMENT OR VERIFY)
1. Smaisika canonicalization:
   - Smaisika is the canonical internal unit (1 Smaisika = 1 USD).
   - Scan codebase for any symbol variants (case-insensitive) and list files where found.
   - Replace usage only in proposed/ with canonical "Smaisika" usage; do not change originals.
   - Provide lib/ledger.js that handles: creditSmaisika(userId, amount, reason), debitSmaisika(userId, amount, reason), convertToSmaisika(amountUSD), convertFromSmaisika(amountSmaisika).

2. Deposits & Withdrawals:
   - Users can deposit USDT or supported crypto into Waides KI custodial wallet. On-chain or exchange-based deposit detection must credit Smaisika.
   - Users can choose NOT to link exchange and instead have their trades executed using Waides KI custodial accounts (pooling option). This must be explicit in UI and terms.
   - Withdrawals are Smaisika → converted to USDT/crypto at admin rate, fees applied, and then send via custodial wallet service. Withdrawals require admin approval unless auto-withdraw enabled and whitelist validated.

3. Linked exchanges:
   - Users can link external exchanges and choose whether trades should execute on their linked exchange (user-controlled keys) or use Waides KI custodial liquidity.
   - For each linked exchange, store encrypted apiKey/apiSecret, withdrawWhitelist[], autoSweepEnabled flag, and apiStatus.
   - Poller must run health checks and set apiStatus: Working | NotWorking | RequiresAction.

4. Trade Execution & Risk:
   - Before any trade the system must:
     a) fetch availableBalance (Smaisika) for that user (custodial or linked)
     b) ask lib/riskManager to compute positionSize based on default 2% or per-user override and botType multipliers
     c) ensure WaidBotPro single trade cap (default 5000 USD)
     d) place trade via chosen connector (or simulate if DEV_MODE)
     e) write trade record to trades.json (traceId, userId, engine, symbol, size, price, result, profitLossSmaisika)
   - If a trade closes with profit, compute profitShare = profit * profitSharingRate (default 0.5). Deduct profitShare from user profit and deposit to admin treasury Smaisika account. Record both transfers in ledger and audit.

5. Profit-Sharing (50/50 default):
   - Configurable admin parameter profitSharingRate (0..1). Default = 0.5 (50%).
   - On each successful trade (realized profit > 0), automatically:
     • debit user profit by profitShare
     • credit adminTreasury by profitShare
     • credit userNetProfit by profit - profitShare
   - Log every step with audit traceId. If using linked exchange & profits happen on exchange, use ledger to represent internal settlement: mark pending admin cut to be swept to admin wallet on approval or auto-sweep.

6. Membership & Pricing (Smaisika)
   - Membership tiers exist (e.g., Bronze, Silver, Gold, Pro). Each tier has a price in Smaisika.
   - Admin can set or change prices via /admin/membership (POST: { tier, priceSmaisika }).
   - Membership perks (higher bot limits, lower profit-sharing fee, faster withdrawals) must be configurable and linked to user profile.
   - When user pays membership: debit Smaisika -> credit treasury -> set user.tier with expiry.

7. Gamification & Referral System
   - Features to implement or verify:
     • Achievements & badges (onboarding, first deposit, 10 trades won, streaks)
     • Daily login streaks & daily tasks (complete tasks for Smaisika rewards)
     • Leaderboards: by profit %, by weekly ROI, by total Smaisika earned
     • Tournaments: sign-up with entry fee in Smaisika, prize pool in Smaisika
     • Social share & copy-trading marketplace (follow top bots or users)
     • Referral tree: track referees and referrer rewards in Smaisika
     • Referral payouts: when referred user earns fees or profits, optionally pay referrer a configurable percentage
     • Gamified UX pages in public/help and user dashboard
   - Provide endpoints for gamification: GET /gamify/leaderboard, POST /gamify/claim, GET /gamify/achievements/:userId

8. Nwaora Chigozie autonomous account
   - Admin-controlled account with its own API keys and treasury.
   - Trades autonomously (binary & spot). Admin can switch engine mode: binary ↔ spot via /admin/nwaora/switch
   - Profits automatically flow to Waides KI treasury and can be reinvested or used as reserves for Smaisika redemptions.

9. API Status & Connector Testing
   - Each connector must expose testConnection() returning { ok:true/false, details }.
   - Build route GET /user/exchanges/:userId/status to show API health per linked exchange.
   - Admin UI should show global connector health dashboard.

10. Manual-Withdraw detection & alerts
   - Poller compares expectedBalance vs observedBalance; if observed < expected - tolerance → create withdrawal-suspect event.
   - Flag user account, pause auto-traders for that user, notify admin, write audit.

11. Scalability & Resilience
   - Use job queues for trade placement, sweep jobs, membership processing, gamify payouts.
   - Background workers (separate processes/tasks) handle heavy operations (polling, reconciliation).
   - Circuit-breaker: if many failures, automatically disable auto-trading globally with admin alert.
   - Use connection pooling for DB; for file store use batched writes and optimistic locking; recommend moving to Postgres/Mongo in production.

12. Security
   - Encrypt api keys at rest using a secure key from env (e.g., ENCRYPTION_KEY). Do not store secrets in plain text.
   - Use admin tokens and MFA for critical admin endpoints (withdraw approvals, profit-sharing changes).
   - Rate-limit sensitive endpoints.
   - Log and rotate audit logs; keep backups of daily ledger and trades.
